# 購物車衝突解決機制

## 概述

本系統實現了完整的樂觀鎖定（Optimistic Locking）機制，用於處理多分頁或多設備同時編輯購物車時的併發衝突問題。

## 工作原理

### 1. 版本追蹤

每個購物車 Session 都有一個版本號（`version`），從 1 開始：
- 每次更新購物車時，版本號遞增
- 客戶端在更新時需提供當前版本號
- 伺服器檢查版本號是否匹配

### 2. 衝突檢測

當發生以下情況時會觸發版本衝突（409 Conflict）：

**場景範例：**
```
時間線：
T1: 分頁 A 讀取購物車 (version=2)
T2: 分頁 B 讀取購物車 (version=2)
T3: 分頁 A 更新購物車 (version=2 → 3) ✅ 成功
T4: 分頁 B 嘗試更新 (version=2 → ?) ❌ 衝突！版本已是 3
```

### 3. 自動衝突解決流程

當檢測到版本衝突時，系統會自動執行以下步驟：

```
1. 保存當前待處理的購物車內容
   ↓
2. 重新從伺服器載入最新購物車 (獲取 version=3)
   ↓
3. 顯示通知：「購物車已被其他分頁更新，正在自動合併您的變更...」
   ↓
4. 執行智能合併邏輯
   ↓
5. 以新版本重試同步 (version=3 → 4)
   ↓
6. 顯示成功通知：「已成功合併！新增了 X 個項目到購物車」
```

### 4. 智能合併策略

系統使用以下邏輯合併購物車項目：

**依據 UUID 匹配：**
- 如果待處理項目的 UUID 在伺服器購物車中存在
  - → 保留伺服器的數量（取最大值）

**依據商品屬性匹配：**
- 檢查 `id`, `size`, `note`, `price` 是否完全相同
  - 存在 → 數量相加
  - 不存在 → 新增該項目

**範例：**
```javascript
伺服器購物車:
[
  { uuid: 'a1', id: 101, name: '牛肉麵', qty: 2, size: '大' }
]

待處理購物車:
[
  { uuid: 'a1', id: 101, name: '牛肉麵', qty: 3, size: '大' },  // UUID 相同
  { uuid: 'b2', id: 102, name: '水餃', qty: 1, size: '小' }    // 新項目
]

合併結果:
[
  { uuid: 'a1', id: 101, name: '牛肉麵', qty: 3, size: '大' },  // 取較大值
  { uuid: 'b2', id: 102, name: '水餃', qty: 1, size: '小' }    // 新增
]
```

## 使用者體驗

### 通知系統

系統會在購物車頁面頂部顯示即時通知：

**處理中（藍色）：**
```
ℹ️ 購物車已被其他分頁更新，正在自動合併您的變更...
```

**成功（綠色）：**
```
✅ 已成功合併！新增了 2 個項目到購物車
```

**失敗（紅色）：**
```
⚠️ 購物車更新失敗，請手動重新整理頁面
```

通知會在 3 秒後自動消失，或可手動關閉。

## 技術實現

### 後端 API

**請求格式：**
```json
PUT /api/cart
{
  "items": [...],
  "note": "備註",
  "version": 5  // 可選：提供時啟用樂觀鎖定
}
```

**成功回應 (200)：**
```json
{
  "items": [...],
  "note": "備註",
  "version": 6
}
```

**衝突回應 (409)：**
```json
{
  "detail": "version_conflict: Cart was modified by another request. Please reload and retry."
}
```

### 前端 Store

**狀態：**
- `cart`: 購物車項目陣列
- `note`: 全域備註
- `version`: 當前版本號
- `conflictNotification`: 通知訊息物件

**關鍵函數：**
- `syncCart()`: 同步購物車，包含衝突處理
- `handleConflictAndRetry()`: 衝突解決邏輯
- `mergeCartItems()`: 智能合併算法

## 測試場景

### 1. 基本衝突測試

**步驟：**
1. 開啟兩個分頁 A 和 B
2. 分頁 A 新增「牛肉麵 x1」
3. 分頁 B 新增「水餃 x1」
4. 切回分頁 A 重新整理頁面

**預期結果：**
- 分頁 A 顯示合併通知
- 購物車包含：牛肉麵 x1、水餃 x1

### 2. 同商品衝突測試

**步驟：**
1. 開啟兩個分頁 A 和 B
2. 分頁 A 新增「牛肉麵（大碗） x2」
3. 分頁 B 新增「牛肉麵（大碗） x3」
4. 等待自動同步完成

**預期結果：**
- 後完成的分頁顯示合併通知
- 購物車包含：牛肉麵（大碗） x5（數量相加）

### 3. 連續衝突測試

**步驟：**
1. 開啟三個分頁 A、B、C
2. 快速依序在各分頁新增不同商品
3. 觀察同步狀況

**預期結果：**
- 所有商品都正確合併
- 最多顯示 1-2 次衝突通知
- 最終版本號反映所有變更次數

## 向後相容性

**不提供版本號時：**
- 系統仍接受更新請求
- 不進行版本檢查
- 適用於舊版客戶端

**範例：**
```json
PUT /api/cart
{
  "items": [...],
  "note": "備註"
  // 沒有 version 欄位 → 總是成功
}
```

## 效能考量

### 自動重試限制

- 衝突時僅重試**一次**，避免無限循環
- 若第二次仍失敗，顯示錯誤通知請使用者手動重新整理

### 同步延遲

- 使用 250ms 防抖（debounce）減少伺服器請求
- 避免每次按鍵都觸發 API 呼叫

### 記憶體佔用

- 衝突處理時會短暫複製購物車內容
- 處理完成後立即釋放臨時資料

## 已知限制

1. **單一分頁/裝置限制**
   - 目前使用 Cookie 綁定 Session
   - 同一瀏覽器的不同分頁共用 Session
   - 無法處理真正的多裝置實時同步（需 WebSocket）

2. **合併策略限制**
   - 數量取較大值可能不符合某些業務邏輯
   - 可依需求調整為「總是相加」或「詢問使用者」

3. **離線支援**
   - 目前無離線快取機制
   - 網路斷線時無法同步

## 故障排除

**問題：版本號一直不匹配**
- 檢查是否有其他分頁/裝置正在修改
- 清除瀏覽器 Cookie 重新開始

**問題：合併結果不符預期**
- 檢查商品 UUID 是否正確產生
- 確認 `mergeCartItems()` 邏輯符合需求

**問題：通知不顯示**
- 檢查 `conflictNotification` 是否正確綁定
- 確認 Vue 元件已正確 import storeToRefs

## 未來改進方向

1. **WebSocket 實時同步**
   - 多裝置即時更新
   - 減少輪詢需求

2. **使用者確認機制**
   - 重大衝突時詢問使用者
   - 提供「保留我的」或「保留伺服器」選項

3. **衝突歷史記錄**
   - 記錄所有衝突事件
   - 供管理員分析使用者行為

4. **更細緻的合併策略**
   - 依商品類型採用不同合併邏輯
   - 支援自訂合併規則
